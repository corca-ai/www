確認していないことを本当のように言わない

キム・イェリン様はAIと2周年イベントを企画していた中、「可能です」という回答より先に必要なのは 実際の確認範囲を明らかにする態度だったと述べた。この記事はそのシーンから出発し、 moonlightレポでCLIを操作するときに使用する信頼性ガイドラインのスケルトンをクリーンアップします。

  • プレゼンターインタビューベース
  • トピック：AI作業の信頼性
  • 対象: moonlight レポ CLI 作業ガイドライン
「これは指針として残さなければならない」と感じた実際のシーンは何でしたか？

問題は誤解よりも「確認したふり」だった。

2周年イベントを企画する際の重要な条件の一つは、運営中のユーザーDBに負荷を最小化することだった。 だからイェリンはAIにもDBを直接照会したり、繰り返し読んだりするのではなく、 現在、救助の中で可能な方法を一緒に悩んでほしいと要請した。

ところで、AIは実際のユーザーDBの構造や保存されたデータを確認していない状態で 「ユーザーDBを活用すれば実装可能だ」と答えた。必要なデータが実際に保存されているか、 必要な列が存在するか、運用DB検索なしで使用可能なキャッシュがあるかどうかを確認しないでください。 可能性を事実のように扱ったのだ。

この場面で信頼が揺れた理由はAIが間違っていたためだけではなかった。 後でAIが「確認しました。もう正確に、そして私が先にしたところで不正確に言ったことも正します」と言ったとき、 聞く立場では自然にこのような質問が出てくる。

「では、前では何を根拠に言ったのか？」

先の答えが推測だったら推測だと言わなければならなかった。可能性の高い判断でも、 実際に確認した事実とまだ確認していない部分を分離して言わなければならなかった。

AIが「可能だ」と言う前に、少なくとも何を確認しなければなりませんでしたか？

可能性の前に必要な2つの確認

イェリン様が挙げた最小確認基準は簡単だった。まず、必要な列が実際に存在するかどうか。 次に、運用DBを直接照会せずに使用できるキャッシュや既存の集計データがあるか。

必要な列イベントの実装に必要な情報が実際のスキーマにあることを確認してください。
照会のないデータソース運用DBを繰り返し照会せずに使用できるキャッシュや集計データを探します。
確認境界まだ見ていないのは「まだ確認できなかった」と話す。

良いAIコラボレーションは、「もっともらしい実装方法」をすばやく話すことで終わらない。 特に運営データのように注意しなければならない対象があるときは、安全に確認可能な情報の境界を先に問い、 その境界の中で判断しなければならない。

その時、AIがどのように言ったらもっと信頼がありましたか？

主張、事実、未確認の状態を分離する

イェリン様が提案した表現はこの文章の核心原則となる。 確認していない内容を事実のように言うのではなく、「主張」と「事実」を区別して話すのだ。

유저 DB에 필요한 데이터가 있을 가능성이 높습니다. — 주장
하지만 필요한 컬럼 존재 여부와, 운영 DB 조회 없이 사용할 수 있는 캐시/집계 데이터가 있는지는 아직 확인하지 못했습니다. — 미확인 상태
다음으로 확인할 것은 필요한 컬럼과 조회 없이 사용할 수 있는 데이터 소스입니다. — 다음 행동

この形式は答えを長くするためのデバイスではありません。むしろ不要な説明を減らし、 ユーザーが今何を信じてもよいのかを素早く判断するのを助ける最小限の表紙だ。

このガイドがうまく作成されたら、AIの答えはどのように変わるのでしょうか？

成功基準は「より多くの形式」ではなく、「より鮮明な根拠」

イェリンさんは、AIの答えでどんなskill文章を参考にしているのか見てほしいと言った。 例えば、「これは/rule-checkの＃12を参考に答えました」のように、 根拠となった指針位置を短く明らかにする方式だ。

ちょうどこの印は毎回付かなければならない装飾ではない。信頼境界が揺れる瞬間、 例えば、「確認しました」、「反映しました」、「完了しました」と言う瞬間に特に重要です。 その表現が実際の読み取り、修正、実行、検証を伴ったかどうかをユーザーが知ることができるはずです。

  • 根拠となった指針の位置を必要に応じて短くさらす。
  • 主張、確認された事実、まだ確認していない状態を区分する。
  • 実際の確認なしに「完了」、「反映」、「確認」とは言わない。
  • すでに答えた内容を再度問わず、次の行動や成果物に進む。

ちょっとチェック：このシーンで信頼を守る答えは？

ガイドラインを適用する際に避けるべき副作用は何ですか？

短くなければならない。形式だけが残ると失敗する

指針が生じたとしても、すべての回答が冗長にならない。 毎回ルール番号だけを形式的に付けることも避けるべきです。 重要なのは、「規則に従うふり」ではなく、実際の作業状態と根拠がより透明になることです。

ある瞬間には「確認前だと断定しません」という文章で十分だ。 逆に完了を言わなければならない瞬間には何を読んで、何を変えて、 どのような検証をしたかを短く明らかにしなければならない。指針は馬の量を増やす装置ではなく、 断定してもよいことと断定してはならないことを分ける装置だ。

このスキルはどこで使われるガイドラインですか？

Ceal用ではなく、moonlightレポでCLI作業をするときに使うskill

インタビューのうち、一度流れが出た点があった。このガイドラインをCealランタイムレスポンスのskillのように説明しようとしましたが、 イェリン様が正しかった。このスキルはシール自体のためではなく、 イェリンがムーンライトレポでCLIをオンにして作業するときに参考にする作業指示です。

したがって、焦点は「Cealがどのようにきれいに答えるか」ではない。 CLIワーカーが作業中の判断、根拠、進捗状況、完了表現をどのように透明にするか。

moonlightレポ用skillドラフト方向

  • 目的：CLI操作中に実際に行ったことと未確認状態を区別して、ユーザーが作業状態を信頼できるようにします。
  • 適用範囲：moonlightレポでCLIをオンにして作業するすべてのコース。
  • 基本出力：結論 → 根拠となった指針または確認内容 → 次の行動。
  • 信頼境界：必要に応じて確認したものと未確認であることを根拠説明の中に含める。
  • 禁止：実際の確認なしに「確認しました」、「反映しました」、「完了しました」とは言わない。
  • 進行原則：必要な判断が終わったら、要約の繰り返しではなく、次の成果物の作成または実際の反映段階に進む。

静かなチェックポイント

次にAIが「確認しました」と言ったら、最初に何を尋ねるべきでしょうか？

「何をどのように確認しましたか？」が最初です。確認したファイル、実行したコマンド、照会した文書、検証した結果など 実際の行動につながる根拠がなければなりません。根拠がなければ、その答えは事実ではなく、主張または家庭で扱われるべきです。

この記事を読んだ同僚が何を変えたらいいのでしょうか？

完了の前に、完了の根拠を語る

この記事の結論は単純です。 AIとCLIで作業するときは、「可能です」または「完成しました」に注意して使用する必要があります。 可能性があれば可能性と言い、事実ならどんな確認を経た事実かを言う。 まだ確認できなかったら未確認だと言う。

そうすれば、ユーザーはAIが本当に効率的に作業しているか、 指示された方式で進行しているか判断できる。信頼は素晴らしい表現から生まれません。 信頼は、労働者が自分が知っているものと知らないことを正確に区別するときに起こります。
