Don't say something you haven't confirmed as fact

Based on the AX sharing session and experience using Ceal, we summarize the working principles and CLI operating rules that prevent AI from saying unconfirmed information as fact.

June 25, 2026 김예린

ceal-terview · 2026-06-25 AX sharing meeting-based internal blog draft

Don't say something you haven't confirmed as fact

While Yerin Kim was planning a 2nd anniversary event with AI, what she needed before the answer was “It is possible.” He said it was an attitude that revealed the actual scope of verification. This article starts from that scene, Organizes the framework of reliability guidelines to be used when performing CLI operations in the moonlight repo.

  • Based on presenter interview
  • Topic: AI task reliability
  • Target: moonlight repo CLI operation instructions
What was the actual scene that made you think, “I need to leave this as a guideline”?

The problem was “pretending to check” rather than a wrong answer.

One of the important conditions when planning the 2nd anniversary event was to minimize the load on the operating user database. So, Yerin does not directly query or repeatedly read the DB for AI. We asked them to consider together possible methods within the current structure.

However, AI does not check the actual user DB structure or stored data. He answered, “It can be implemented by using the user DB.” whether the required data is actually stored; Without checking whether the required column exists or whether there is a cache that can be used without querying the operational DB Possibilities were treated as if they were facts.

The reason trust was shaken in this scene was not just because the AI was wrong. Later, when the AI says, “I checked. Now I’m correct, and I’ll also correct what I said incorrectly in one place earlier.” As a listener, these questions naturally arise.

“Then what did you base your statement on earlier?”

If the previous answer was a guess, you should have said it was a guess. Even if it is a highly probable decision, I had to separate the facts that were actually confirmed from the parts that had not yet been confirmed.

What should we have at least checked before saying AI is “possible”?

Two things you need to know before you know it's possible

The minimum verification standard chosen by Yerin was simple. First, whether the required column actually exists. Second, is there cache or existing aggregate data that can be used without directly querying the operational DB?

required columnCheck whether the information necessary for event implementation is in the actual schema.
Data source without lookupFind cache or aggregate data that can be used without repeatedly querying the operational DB.
check borderIf you haven't seen it yet, say "I haven't confirmed it yet."

Good AI collaboration doesn’t end with quickly talking about “plausible implementation methods.” Especially when there is something that requires careful attention, such as operational data, we first ask about the boundaries of information that can be safely verified. Judgment must be made within those boundaries.

What would the AI have said back then to make it more trustworthy?

Separate claims, facts, and unknowns

The expression proposed by Yerin becomes the core principle of this article. Do not speak of unconfirmed information as if it were fact, but distinguish between “claims” and “facts.”

유저 DB에 필요한 데이터가 있을 가능성이 높습니다. — 주장
하지만 필요한 컬럼 존재 여부와, 운영 DB 조회 없이 사용할 수 있는 캐시/집계 데이터가 있는지는 아직 확인하지 못했습니다. — 미확인 상태
다음으로 확인할 것은 필요한 컬럼과 조회 없이 사용할 수 있는 데이터 소스입니다. — 다음 행동

This format is not a device for making the answer longer. Rather, reduce unnecessary explanations, It is a minimal sign that helps users quickly decide what they can trust now.

If this guide is well-made, how should AI answers change?

The criterion for success is not “more form” but “clearer evidence”

Yerin said that it would be nice to be able to see which skill sentence is being referenced in the AI answer. For example, “This/rule-checkAs in, “I answered by referring to #12 of .” It is a method of briefly revealing the location of the guideline on which it is based.

However, this sign is not a decoration that must be attached every time. The moment the boundaries of trust can be shaken, This is especially important in the moments when you say, for example, “I checked,” “I reflected,” or “I’m done.” The user must be able to know whether the expression was accompanied by actual reading, modification, execution, or verification.

  • Briefly reveal the location of the guidance on which it is based when necessary.
  • Distinguish between claims, confirmed facts, and unconfirmed status.
  • Do not say “complete,” “reflect,” or “confirm” without actual confirmation.
  • Rather than asking questions again about what has already been answered, proceed to the next action or product.

Quick check: What answer preserves trust in this scene?

What side effects must be avoided when applying the guidelines?

It must be short. If only the format remains, it is a failure.

Just because guidelines are in place, not all answers should become lengthy. You should also avoid formally adding just a rule number each time. The important thing is not to “pretend to follow the rules,” but to make the actual work status and rationale more transparent.

At some moments, one sentence, “I won’t make any conclusions because it’s not confirmed,” is enough. Conversely, at the moment when you have to say it's complete, what did you read, what did you change, You must briefly disclose what verification was done. Guidelines are not a device to increase the amount of speech; It is a device that distinguishes between what can be neat and what cannot be neat.

What is this skill used for?

Not for Ceal, but a skill used when working with CLI in the moonlight repo

There was one point during the interview where the flow went astray. I tried to explain this guideline as a skill for Ceal runtime response, but Yerin corrected it. This skill is not for Ceal itself, These are the work instructions that Yerin will refer to when turning on the CLI and working in the moonlight repo.

Therefore, the focus is not “How will Ceal respond nicely?” The issue lies in how CLI workers transparently express judgment, rationale, progress status, and completion during work.

Skill draft direction for moonlight repo

  • Purpose:It distinguishes between actual performance and unconfirmed status during CLI operations, allowing users to trust the status of the operation.
  • Scope of application:The whole process of turning on and working with the CLI in the moonlight repo.
  • Default output:Conclusion → Guidelines or confirmations based on → Next action.
  • Trust boundary:If necessary, include confirmed and unconfirmed information in the explanation of the evidence.
  • Prohibited:Do not say “I have confirmed,” “I have reflected,” or “I have completed” without actual confirmation.
  • Progress principles:Once the necessary judgments have been made, move on to the next step of creating the output or actually reflecting it, rather than repeating the summary.

quiet checkpoint

The next time an AI says, “I got it,” what should you ask first?

The first question is “What did you check and how?” Like files checked, commands executed, documents viewed, and results verified. There must be a rationale that leads to real action. If there is no basis, the answer should be treated as a claim or assumption rather than a fact.

What should colleagues who read this do differently?

Before completion, state the basis for completion

The conclusion of this article is simple. When working with AI and CLI, you need to be more careful when using “I can” or “I’ve done it.” If there is a possibility, say it is a possibility, and if it is true, say what kind of confirmed fact it is. If it has not been confirmed yet, it is said to be unconfirmed.

That way, users can check whether the AI is really working efficiently, You can determine whether you are proceeding in the manner instructed. Trust does not come from fancy expressions. Trust arises when workers clearly distinguish between what they know and what they do not know.
