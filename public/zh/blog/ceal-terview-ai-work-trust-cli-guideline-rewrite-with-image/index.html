不要说一些你尚未证实的事实的话

基于AX分享会和Ceal的使用经验，我们总结了防止AI将未经证实的信息说成事实的工作原理和CLI操作规则。

2026年6月25日 김예린

ceal-terview · 2026-06-25 基于AX共享会议的内部博客草案

不要说一些你尚未证实的事实的话

当 Yerin Kim 计划与 AI 一起举办两周年活动时，她需要的是“这是可能的”。 他说，这是一种态度，揭示了核查的实际范围。这篇文章就从那个场景开始， 组织在 Moonlight 存储库中执行 CLI 操作时使用的可靠性指南框架。

  • 根据主持人采访
  • 主题：AI任务可靠性
  • 目标：moonlight repo CLI操作说明
实际场景是什么让您想到“我需要将此作为指导”？

问题在于“假装检查”而不是错误的答案。

规划两周年活动时的重要条件之一是尽量减少操作用户数据库的负载。 因此，Yerin 不会直接查询或重复读取 AI 的 DB。 我们要求他们一起考虑当前结构内可能的方法。

然而，AI 不会检查实际的用户数据库结构或存储的数据。 他回答：“可以通过用户DB来实现。”是否实际存储了所需的数据； 无需检查所需列是否存在或是否有可以使用的缓存，无需查询操作DB 可能性被视为事实。

在这个场景中信任动摇的原因不仅仅是因为人工智能是错误的。 后来，当AI说：“我检查过了，现在我是对的，我也会纠正我之前说错的地方。” 作为倾听者，自然会产生这些问题。

“那你刚才的说法有何根据？”

如果前面的答案是猜测，你就应该说这是猜测。即使这是一个可能性很大的决定， 我必须将实际已确认的事实与尚未确认的部分分开。

在说人工智能“可能”之前，我们至少应该检查什么？

在你知道这是可能的之前你需要知道两件事

艺琳选择的最低验证标准很简单。首先，所需的列是否确实存在。 其次，是否有缓存或现有聚合数据可以在不直接查询操作数据库的情况下使用？

必填栏检查事件实现所需的信息是否在实际模式中。
无需查找的数据源查找可以使用的缓存或聚合数据，而无需重复查询操作数据库。
检查边界如果您还没有看到，请说“我还没有确认”。

良好的人工智能协作并不以快速谈论“合理的实施方法”而结束。 尤其是当有一些需要仔细关注的东西时，例如操作数据，我们首先询问可以安全验证的信息的边界。 必须在这些界限内做出判断。

那时人工智能会说些什么来使其更值得信赖？

单独的主张、事实和未知数

Yerin提出的表述成为本文的核心原则。 不要将未经证实的信息当作事实来谈论，而要区分“主张”和“事实”。

유저 DB에 필요한 데이터가 있을 가능성이 높습니다. — 주장
하지만 필요한 컬럼 존재 여부와, 운영 DB 조회 없이 사용할 수 있는 캐시/집계 데이터가 있는지는 아직 확인하지 못했습니다. — 미확인 상태
다음으로 확인할 것은 필요한 컬럼과 조회 없이 사용할 수 있는 데이터 소스입니다. — 다음 행동

这种格式并不是为了使答案更长。而是减少不必要的解释， 这是一个最小的标志，可以帮助用户快速决定他们现在可以信任什么。

如果这个指南制作精良，AI 答案应该如何改变？

成功的标准不是“形式更丰富”，而是“证据更清晰”

Yerin 表示，如果能够看到 AI 答案中引用了哪个技能句子，那就太好了。 例如，“这/rule-check例如，“我通过参考第 12 条来回答。” 它是一种简要揭示其所依据的指南位置的方法。

然而，这个标志并不是每次都必须贴上的装饰。当信任的界限被动摇的那一刻， 当你说“我检查了”、“我反思了”或“我完成了”时，这一点尤其重要。 用户必须能够知道该表达式是否伴随着实际的读取、修改、执行或验证。

  • 必要时简要透露其所依据的指南的位置。
  • 区分主张、已确认的事实和未确认的状态。
  • 未经实际确认，请勿说“完成”、“反映”或“确认”。
  • 不要再次询问已经回答过的问题，而是继续执行下一个操作或产品。

快速检查：什么答案可以保持对这个场景的信任？

应用指南时必须避免哪些副作用？

一定要短。如果只剩下格式，那就是失败。

仅仅因为指导方针已经到位，并不是所有的答案都应该变得冗长。 您还应该避免每次都正式添加规则编号。 重要的不是“假装遵守规则”，而是让实际工作状况和理由更加透明。

有时候，一句话“我不会下任何结论，因为还没有得到证实”就足够了。 相反，当你不得不说它完成的那一刻，你读了什么，你改变了什么， 您必须简要披露进行了哪些验证。指导方针不是增加讲话量的手段；而是一种手段。 它是一种区分什么可以整洁和什么不能整洁的装置。

这个技能是用来做什么的？

不适用于 Ceal，而是在 Moonlight 存储库中使用 CLI 时使用的技能

采访过程中有一处流程出现了偏差。我试图将此指南解释为 Ceal 运行时响应的技能，但是 艺琳纠正了它。这个技能并不是针对Ceal本身的， 这些是 Yerin 在打开 CLI 并在 Moonlight 存储库中工作时将参考的工作说明。

因此，焦点不是“卡尔将如何应对？” 问题在于 CLI 工作人员如何在工作中透明地表达判断、理由、进展情况和完成情况。

月光仓库的技能草案方向

  • 目的：它区分 CLI 操作过程中的实际性能和未确认的状态，使用户可以信任操作的状态。
  • 适用范围：在 Moonlight 存储库中打开并使用 CLI 的整个过程。
  • 默认输出：结论 → 基于 → 下一步行动的指南或确认。
  • 信任边界：如有必要，在证据解释中包括已确认和未确认的信息。
  • 禁止：未经实际确认，请勿说“我已确认”、“我已反映”或“我已完成”。
  • 进度原则：一旦做出了必要的判断，就继续下一步，即创建输出或实际反映输出，而不是重复总结。

安静的检查站

下次当人工智能说“我明白了”时，你应该首先问什么？

第一个问题是“你检查了什么以及如何检查？”就像检查文件、执行命令、查看文档和验证结果一样。 必须有一个导致实际行动的理由。如果没有依据，答案应被视为主张或假设，而不是事实。

阅读本文的同事应该采取哪些不同的做法？

完成前，说明完成的基础

这篇文章的结论很简单。使用 AI 和 CLI 时，在使用“我可以”或“我已经做到了”时需要更加小心。 如果有可能性，就说它是一种可能性，如果是真的，就说它是哪种已证实的事实。 如果尚未得到证实，则称其为未证实。

这样，用户可以检查人工智能是否真正高效地工作， 您可以确定是否按照指示的方式进行。信任并非来自花哨的表达。 当员工清楚地区分他们所知道的和他们不知道的事情时，信任就会产生。

出版须知：此草案供内部共享。提供了与操作数据库、用户数据和内部存储库结构相关的详细信息。 向外界披露时，将信息概括化或匿名化会更安全。
